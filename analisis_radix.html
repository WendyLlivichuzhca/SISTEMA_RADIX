🔴 Bugs críticos — Rompen la lógica del plan

Monto incorrecto para el salto a Fase 1 — solo se reservan $30 en vez de $100

Según el plan, al completar el Tablero C, el sistema debe reservar automáticamente $100 para el acceso a Fase 1. Sin embargo, en matrix_logic.php línea 63, el código solo registra $30. Esto hace que el salto a Fase 1 nunca tenga los fondos correctos.

📋 PLAN Deducción al completar Tablero C: -$100 → Salto a Fase 1 (Clase 1) -$10 → Re-entrada Fase 0 -$10 → Tesorería RADIX ───────────────────────── Neto: $40 líquido al usuario
💻 CÓDIGO (matrix_logic.php, línea 63) // B. Reserva Salto Fase 1 ($30) ← INCORRECTO $stmt->execute([$usuario_id, 0, 30.00, ← Debería ser 100.00 'salto_fase_1', 'completado']);

Los $40 netos nunca se envían al usuario — no existe el mecanismo de pago

El plan promete $40 USDT líquidos directo al bolsillo del usuario al completar Fase 0. En el código, los sobrantes de Tablero A ($10) y Tablero B ($20) se registran como pagos a favor del patrocinador del usuario (líneas 41–49 de matrix_logic.php), no al usuario mismo. Tampoco existe ningún endpoint ni función que calcule y envíe el pago neto a la wallet del usuario.

📋 PLAN Sobrante A: $10 → usuario guarda Sobrante B: $20 → usuario guarda Sobrante C: $120 - deducciones Neto final: $40 USDT al usuario
💻 CÓDIGO (matrix_logic.php, líneas 41–49) // Al avanzar A→B se paga $20 al // PATROCINADOR, no al usuario $stmt->execute([ $usuario_id, $patrocinador_id, // ← Incorrecto $monto_regalo, 'regalo', 'pendiente' ]);

schema.sql está desactualizado — ejecutarlo destruye la compatibilidad con el código

El archivo database/schema.sql es la versión 1.0 y le faltan 4 elementos que el código actual (v2.0) necesita obligatoriamente. Si alguien ejecuta este archivo en producción, la app fallará con errores de "columna no encontrada".

❌ Faltan en schema.sql — el código los necesita 1. Tabla sistema_config — usada en clon_logic.php para tesorería (schema.sql solo tiene la vieja tabla 'configuracion') 2. Columna usuarios.tipo_usuario ENUM('real','clon') — usada en clon_logic.php al crear clones 3. Columna tableros_progreso.ciclo INT DEFAULT 1 — usada en matrix_logic.php y clon_logic.php 4. Tipos de pago en ENUM: 'tesoreria', 'salto_fase_1' — usados en matrix_logic.php, no existen en el ENUM de schema.sql 5. pagos.id_receptor FK a usuarios — el código inserta id_receptor=0 para la tesorería, rompiendo la FK constraint

✅ Buenas noticias: db_setup.php tiene las migraciones correctas. El problema es que schema.sql no fue actualizado para reflejar la v2.0.

admin_actions.php no tiene ninguna verificación de autenticación

El endpoint radix_api/admin_actions.php que permite cambiar el estado de cualquier usuario (suspender, activar, etc.) no verifica si el solicitante es un administrador. Cualquier persona que conozca la URL puede enviar un POST y modificar usuarios arbitrariamente.

🟠 Advertencias — Afectan el funcionamiento

stats.php cuenta clones como usuarios reales — infla las métricas

En stats.php, la query cuenta todos los usuarios con estado = 'activo', sin excluir los que tienen tipo_usuario = 'clon'. El dashboard mostrará más "usuarios" de los que realmente hay, lo cual distorsiona las métricas de crecimiento que el plan menciona en la tabla de ejemplos (1 usuario, 2 usuarios, 4 usuarios, etc.).

Fix: Agregar AND tipo_usuario = 'real' a la query de stats.

admin_data.php (tab "config") usa la tabla vieja — siempre falla

Cuando el admin abre la pestaña de configuración, admin_data.php ejecuta SELECT clave, valor FROM configuracion. Esta tabla fue reemplazada por sistema_config en la v2.0. Si la tabla vieja no existe en producción, el tab arroja error 500. Si existe, muestra datos obsoletos (monto_regalo=$5, fee=$1) que ya no controlan nada.

Cálculo de ganancias en user_data.php puede ser incorrecto

La fórmula de user_data.php calcula ganancias como recibido de regalos − pagado como emisor. El problema: los pagos de tesorería, salto_fase_1 y reentrada se restan como si el usuario los hubiera "pagado" de su bolsillo, cuando en realidad son deducciones automáticas del sistema. El resultado mostrado en el dashboard puede no reflejar el $40 neto que promete el plan.

Un clon puede ser usado como patrocinador al registrar nuevos usuarios

En registro.php, al recibir un patrocinador_wallet, el código solo verifica que no sea la misma wallet del usuario (anti-auto-patrocinio). No valida si el patrocinador es un clon (tipo_usuario = 'clon'). Según el plan, los clones son entidades artificiales y no deberían poder reclutar personas reales.

No hay confirmación de pagos reales — todo queda en "pendiente"

Los pagos creados al registrar nuevos usuarios quedan con estado = 'pendiente'. No existe ningún webhook, endpoint o proceso automático que los confirme. El campo hash_transaccion existe en el schema pero nunca se usa. Sin este puente con blockchain, la app asume que todos pagan sin verificarlo.

🔵 Riesgos de seguridad

Credenciales de base de datos en texto plano en config.php

El archivo radix_api/config.php contiene el usuario y contraseña de la base de datos escritos directamente. Si este archivo es accesible desde el servidor web o se sube a un repositorio público, las credenciales quedan expuestas. Considera usar variables de entorno (.env) o un archivo de config fuera del directorio público.

CORS completamente abierto — cualquier dominio puede llamar la API

En config.php, el header Access-Control-Allow-Origin: * permite que cualquier sitio web en internet haga peticiones a tu API. En producción, esto debe restringirse solo al dominio de tu frontend.

Las firmas de wallet se capturan pero nunca se validan

En registro.php, se reciben los campos signature y message del usuario y se guardan en el log de auditoría, pero nunca se verifica criptográficamente que la firma corresponda a la wallet. Esto significa que cualquiera puede registrarse con la wallet de otra persona sin necesitar la clave privada.
