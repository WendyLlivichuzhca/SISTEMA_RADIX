🔴 Bugs críticos — Rompen la lógica

Se activan 2 clones en vez de 1 por tablero

Cuando el usuario completa el Tablero C, matrix_logic.php llama a intentarActivarClon() al final (línea 104). Pero clon_logic.php también la llama después de crear el primer clon. Resultado: se crean 2 clones seguidos para el mismo usuario (IDs 1016 y 1017 en tu BD). El diagrama dice 1 clon + 2 reales por tablero.

Fix: Agregar una condición que verifique que el usuario no tenga ya un clon asignado en el ciclo actual antes de crear otro.

La wallet se pasa en la URL del dashboard — cualquiera puede ver el panel de otro

app.js redirige a dashboard.php?wallet=0x... y dashboard.php acepta esa wallet sin verificar si pertenece a quien la solicita. Cualquier persona que conozca la wallet de otro usuario puede entrar a su dashboard directamente escribiendo la URL.

Fix: Usar sesiones PHP ($_SESSION) para guardar la wallet autenticada. No pasarla nunca en la URL.

user_data.php llama a verificarAvanceTablero() y intentarActivarClon() en cada carga del dashboard

Cada vez que el usuario abre su dashboard, el servidor ejecuta las funciones de avance y de clon. Si el dashboard se recarga múltiples veces en pocos segundos, podría disparar múltiples clones o avances duplicados. Estas funciones deben ejecutarse solo cuando ocurre un evento real (nuevo referido registrado).

🟠 Advertencias — Afectan la experiencia

La conexión de wallet usa prompt() — no hay integración real con MetaMask/SafePal

app.js usa prompt() para pedir la wallet manualmente. No hay integración con Web3.js ni ethers.js. Esto significa que cualquier persona puede escribir la wallet de otro y registrarse como si fuera ella. Para producción necesitas: window.ethereum.request({ method: 'eth_requestAccounts' }).

Los pagos entre usuarios quedan en "pendiente" — no hay verificación blockchain

Cuando un usuario se registra, el pago de $10 se crea como estado = 'pendiente'. No existe ningún mecanismo que confirme si el pago realmente ocurrió en la blockchain. El sistema asume que todos pagan. En producción, necesitas un listener de eventos del smart contract o un webhook de pago.

stats.php cuenta clones como usuarios reales

La query de estadísticas no filtra tipo_usuario = 'real'. El home mostraría 17 usuarios cuando en realidad solo hay 15 reales (más 2 clones más el sistema).

Sub-usuarios de prueba (1007-1015) sin patrocinador_id

Los 9 usuarios de prueba en la BD tienen patrocinador_id = NULL. En producción cada persona debe entrar con un referido. Estos datos de prueba deberían limpiarse antes del lanzamiento.

dashboard.php mezcla PHP y HTML sin plantilla — difícil de mantener

Todo el CSS, HTML y JS está en un solo archivo PHP de más de 300 líneas. Si necesitas cambiar el diseño en el futuro, es muy difícil de editar. Separar en archivos dedicados haría el código mucho más mantenible.

🔵 Seguridad — Urgente para producción

Credenciales de BD en texto plano en config.php

config.php contiene usuario y contraseña de la BD visibles directamente. Si el repositorio es público o el archivo se expone por error, las credenciales quedan comprometidas. Usar variables de entorno o un archivo .env fuera del directorio web.

admin_actions.php accesible sin autenticación

Cualquier persona puede hacer un POST a este endpoint y cambiar el estado de cualquier usuario. No hay verificación de sesión admin. Agregar session_start() y verificar el rol antes de ejecutar cualquier acción.

Las firmas de wallet se guardan pero nunca se verifican

registro.php guarda signature y message en auditoría pero no verifica criptográficamente que la firma pertenezca a esa wallet. Cualquiera puede registrar la wallet de otro sin tener la clave privada.

Archivos de debug/test accesibles públicamente

Existen archivos como debug_levels.php, test_matrix.php, simulate_jump_c.php, reset_db.php y fix_db.php en el directorio público. Estos archivos pueden exponer información sensible o permitir que alguien reinicie la base de datos desde el navegador.

🟢 Mejoras funcionales recomendadas

Integrar MetaMask / SafePal real con ethers.js

Reemplazar el prompt() con una conexión real a la billetera del usuario. Esto verifica automáticamente la propiedad de la wallet y añade una capa de seguridad real. Código base:

const accounts = await window.ethereum.request(
  { method: 'eth_requestAccounts' }
);
const wallet = accounts[0]; // Wallet real verificada

Sistema de notificaciones en tiempo real

Cuando alguien se une usando tu link de referido, el usuario no recibe ninguna notificación. Implementar WebSockets o polling cada 30 segundos para notificar eventos como: "¡Nuevo referido! P1 se unió a tu red", "Tablero A completado, avanzaste a B", "🤖 Clon activado para ti".

Visualización del árbol de red en el dashboard

El dashboard actual muestra los referidos como lista. Un árbol visual interactivo (como el diagrama que tú tienes) haría la experiencia mucho más atractiva y fácil de entender para los usuarios. Librerías sugeridas: D3.js o vis.js (ambas gratuitas).

Página de retiro / historial de ganancias

El dashboard muestra las ganancias calculadas pero no hay ninguna página donde el usuario pueda solicitar el retiro de sus $40. Necesitas una sección de "Mis Ganancias" con: saldo disponible, historial de movimientos (A, B, C) y botón de solicitar retiro.

Link de referido propio más visible y copiable

El link ?ref=0x... ya existe en user_data.php pero no hay un botón visible en el dashboard para copiarlo con un click. Es la funcionalidad más importante del sistema — debe estar destacada con un botón de copiar y un contador de "Referidos invitados: 2/3".

Email / Telegram de notificación al completar tablero

Cuando el usuario completa un tablero y avanza, no recibe ninguna notificación fuera de la app. Integrar un sistema de notificaciones (Telegram Bot API es el más fácil) para avisar: "¡Completaste Tablero A! Tus $10 están listos."

Límite de 1 clon por tablero por usuario (fix del bug + mejora)

Corregir el bug de 2 clones y además agregar un contador visible en el dashboard: "Slots: 2/3 llenos (1 real, 1 clon)" para que el usuario sepa exactamente qué le falta para avanzar.

Panel admin más completo con control de Tesorería

El admin actual solo muestra stats básicas. Agregar: balance en tiempo real de la Tesorería, historial de clones activados, usuarios por tablero, gráfica de crecimiento diario, y botón manual de "Activar Clon" para casos de emergencia.

🎨 Mejoras de diseño recomendadas

Landing page (index.html) — Agregar sección de ROI visual

El landing actual explica el sistema en texto. Añadir una sección visual con el diagrama de tableros A→B→C animado, mostrando cómo el $10 se convierte en $150. Esto aumenta drásticamente la conversión de visitantes a registros. El diagrama que tú tienes en Excel podría ser esa sección.

Mobile responsive — dashboard.php no es responsive

El dashboard usa grid-template-columns: repeat(3, 1fr) sin media queries. En celular los 3 widgets se apilan deformes. La mayoría de usuarios probablemente accederán desde el celular. Agregar al menos: @media(max-width:768px) con columna única.

Animación de progreso de tablero más visual

El dashboard tiene una línea de progreso A→B→C pero es muy simple. Mejorar con: nodos animados que pulsan cuando están activos, contador de "2/3 referidos" en cada nodo, y confetti/animación cuando se completa un tablero.

Estados de los referidos más claros visualmente

Actualmente los referidos se muestran como lista básica. Mejorar con avatares de colores (verde=real activo, azul=clon, gris=pendiente), indicador de en qué tablero está cada referido, y chip de "Pagado ✅" vs "Pendiente ⏳".

Página de onboarding para nuevos usuarios

Cuando un usuario se registra por primera vez, entra directo al dashboard sin ninguna explicación. Una pantalla de bienvenida de 3 pasos (¿Qué es RADIX? → Cómo funciona → Tu primer paso) mejoraría mucho la retención y reduciría confusión.
